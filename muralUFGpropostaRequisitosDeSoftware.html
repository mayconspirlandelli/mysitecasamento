Mur-Uso-ObtUsuGrup - Obter usuários contidos em um GrupoDestinatario

Descrição

A CentralUFG recebe uma requisição para responder o conjunto de usuários e outros grupos que compõem um GrupoDestinatario, realiza a pesquisa e responde a requisição com os usuários e grupoDestinatarios obtidos.

Atores, interesses e frequência de uso

  • Sistema produtor: conhecer os destinatários (usuário e outros grupos de destinatários) relacionados a um determinado GrupoDestinatário.

Frequência típica: diariamente 300 vezes.

Atores secundários e interesses da CentralUFG

  • CAS: fornecer validação de credenciais de usuário produtor de informação.

Precondições

  • CAS ativo.

Sequência Típica

  1. O sistema produtor requisita a CentralUFG os usuários de um determinado GrupoDestinatario.
  2. CentralUFG solicita validação de credencial junto ao CAS.
  3. CAS valida credencial informada.
  4. CentralUFG valida o GrupoDestinatario informado.
  5. CentralUFG verifica se o usuário tem autorização de acesso os usuários deste GrupoDestinatario.
  6. CentralUFG pesquisa os usuários vinculados ao GrupoDestinatario informado.
  7. CentralUFG responde ao sistema produtor os usuários ou outros grupos de destinatário.
  8. Caso de uso se encerra.

Mur-Uso-EmitRelSolDiv - Emitir relatório sobre solicitações de divulgação

Descrição

Um sistema administrador requisita a CentralUFG dados sobre solicitações de divulgação de informação. A CentralUFG consulta as solicitações realizadas segundo critérios definidos na requisição. Cada solicitação de divulgação contida no relatório deve conter:

  • identificador da solicitação
  • nome do solicitante
  • data de abertura
  • hora da abertura
  • estado
  • descrição do estado

Atores, interesses e frequência de uso

  • Sistema administrador: Obter informações para acompanhar o bom andamento da CentralUFG.

Frequência típica: 1 vez por dia.

Precondições

  • Credencial de usuário administrador validada.
  • Seguintes critérios de pesquisa informados:
  • Período que compreende a realização das solicitações de divulgação.
  • Estados das solicitações.

Sequência Típica

  1. O sistema administrador requisita a CentralUFG dados das solicitações de divulgação.
  2. A CentralUFG valida os critérios de pesquisa informados.
  3. A CentralUFG pesquisa todas as solicitações de divulgação com as informações descritas anteriormente, de acordo com os critérios informados.
  4. A CentralUFG responde ao sistema administrador com um conjunto de solicitação de divulgação com as informações descritas anteriormente.
  5. Caso de uso se encerra.

Exceções da Sequência Típica

Tratamento da exceção

Retorno ao passo

2

Critério de pesquisa inválidos.

CentralUFG responde com codigo 20003.

Mur-Uso-HabDispRecNot - Habilitar dispositivo android para receber notificações

Descrição

Para habilitar-se para recebimento de notificações, o dispositivo android envia à CentralUFG o seu identificador (obtido junto ao serviço de mensagens nas nuvens) e opcionalmente as credenciais de um usuário. A CentralUFG registra o dispositivo para receber notificações e adicionalmente, se as credenciais forem informadas e validadas, associa o dispositivo android ao usuário informado.

Fontes de requisito

  • ReqSis10
  • ReqSis11
  • ReqSis12
  • ReqSis19
  • ReqSis20
  • ReqSis21

Atores, interesses e frequência de uso

  • Dispositivo android: Habilitar-se para receber notificações. Frequência típica: Normalmente 10 vez por dia. A cada 6 meses esse número sobe para 300 vezes ao dia durante 2 semanas.

Atores secundários e interesses da CentralUFG

  • CAS: fornecer validação de credenciais de usuário produtor de informação.

Precondições

  • Solicitação realizada por cliente de dispositivos android.

Sequência típica

  1. Dispositivo android requisita se habilitar para receber notificações junto a CentralUFG.
  2. CentralUFG habilita dispositivo android.
  3. CentralUFG solicita validação das credenciais junto ao CAS.
  4. CAS valida a credencial.
  5. CentralUFG desassocia o dispositivo android vinculado anteriormente ao usuário.
  6. CentralUFG associa o dispositivo android com o usuário informados.
  7. CentralUFG responde código [10000]
  8. Caso de uso se encerra.

Mur-Uso-DesabRecNot - Desabilitar dispositivo android para receber notificações dirigidas

Descrição

Para desabilitar o envio de notificações dirigidas para dispositivo android, o dispositivo envia à CentralUFG o seu identificador (obtido junto ao serviço de mensagens nas nuvens) e a credencial do usuário que esta habilitado no respectivo dispositivo. A CentralUFG desabilita o recebimento de notificações dirigidas, contudo, o dispositivo continua recebendo notificações públicas.

Fontes de requisito

  • ReqSis10
  • ReqSis11

  • ReqSis20

  • Identificador do dispositivo informado.

  2. CentralUFG localiza o dispositivo informado.

  4. CentralUFG solicita a validação da credencial informada junto ao CAS.

  6. CentralUFG desabilita envio do usuário informado para o respectivo dispositivo android.

Mur-Uso-AtuOrgPos - Atualizar órgãos da UFG e sua respectiva posição geográfica

Descrição

O sistema administrador solicita à CentralUFG a atualização de órgãos da UFG e a respectiva posição geográfica desses órgãos. A CentralUFG realiza a atualização conforme operação informada.

Essa informação é útil quando um dispositivo android solicita auxilio para deslocamento até um órgão da UFG e a CentralUFG precisa informar a posição geográfica do órgão solicitado.

Fonte do requisito

  • Termo de Abertura – Projeto Mural UFG

  • Requisitos de sistema: reqSis1, reqSis3, reqSis15

Atores, interesses e frequência de uso

  • Sistema administrador: Deseja cadastrar, remover ou alterar um órgão da UFG com sua posição geográfica.
  • Frequência típica: 1 vez por mês.

Precondições

  • Órgão informado.
  • Operação informada(Incluir, remover ou alterar)

Sequência Típica

  1. CentralUFG valida credenciais de usuário administrador.
  2. CentralUFG verifica operação informada pelo sistema administrador :
  1. Se operação for incluir - Sequência Especifica 01
  2. Se operação for remover - Sequência Especifica 02
  3. Se operação for alterar - Sequência Especifica 03
  1. CentralUFG efetiva operação e responde código [20000], caso não seja possível responde código [20001]
  2. Caso de uso se encerra.

Sequência Específica 01

  1. CentralUFG prepara um novo órgão.
  2. Se há informações de DadosContato
  1. prepara novo DadosContato e associa ao novo órgão.
  1. Se há informações de Posicao
  1. prepara nova Posicao e associa ao novo órgão        

Sequência Específica 02

  1. CentralUFG busca órgão, caso não exista responde código [20001] e caso de uso encerra.
  2. Se há informações de DadosContato
  1. Prepara remoção de DadosContato associado ao órgão
  1. Se há informações de Posicao
  1. Prepara remoção Posicao associada ao órgão.
  1. Prepara remoção órgão

 

Sequência Específica 03

  1. CentralUFG busca órgão, caso não exista responde código [20001] e caso de uso encerra.
  2. Altera informações do órgão
  3. Se há informações de DadosContato
  1. Altera DadosContato associado ao órgão
  1. Se há informações de Posicao
  1. Altera Posicao associada ao órgão.

Mur-Uso-ConsPublicacao - Consultar publicações

Descrição

O dispositivo android realiza uma solicitação de consulta de publicações à CentralUFG, que realiza a pesquisa conforme critérios informados e responde a solicitação.

Requisitos de qualidade

  • Confiabilidade: estar disponível 24hs por dia, 7 dias por semana.
  • Eficiência: Capaz de processar 100 requisições por segundo.

Fonte do requisito

  • Termo de Abertura – Projeto Mural UFG
  • Requisitos de sistema: reqSis9

Atores, interesses e frequência de uso

  • Dispositivo android: tem interesse em conhecer as publicações divulgadas. Frequência: 2000 vez ao dia.

Precondições

  • Seguintes critérios de pesquisa informados:
  • Termo: palavra(s) contida(s) na publicação.
  • Período: Intervalo de datas que as publicações estiveram vigentes.

Sequência Típica

  1. CentralUFG verifica termo e período informados pelo dispositivo android.
  2. Central UFG muda estado da requisição para Aceita, se período informado são datas validas, caso contrário Fluxo Alternativo 01.
  3. Central UFG faz busca na sua base por todas as divulgações do tipo Publicação que contenham o termo e que estejam com vigência compreendida no período informado.
  4. CentralUFG envia resultados que atenderam aos critérios definidos na consulta, caso contrário Sequência específica 01.
  5. Central UFG responde a solicitação com código [10000].
  6. Caso de uso se encerra.

Sequência específica 01

  1. Central UFG prepara mensagem de resposta.
  2. Central UFG envia uma mensagem murMens-FalhaResulResp ao Dispositivo.

Fluxo Alternativo 01

  1. Central UFG muda estado da requisição para negada.
  2. Registra a descricaoEstado da requisição com o motivo da negação.
  3. Central responde código [10004] e caso de uso se encerra.

Mur-Uso-AnexArquivo – Anexar arquivo à requisição

Descrição

CentralUFG recebe de um sistema produtor pedido para anexar arquivo à solicitação, valida informações do arquivo anexado (tamanho e tipo) e devolve uma referência a ser usada na requisição.

Requisitos de qualidade

  • Funcionalidade:capaz de armazenar arquivos anexados de forma segura (somente pessoas com autorização tenham acesso).

  • Confiabilidade: garantia que referências dos arquivos serão sempre válidas.

Fonte do requisito

  • Termo de Abertura – Projeto Mural UFG

  • Requisitos de sistema: reqSis8, reqSis19

Atores, interesses e frequência de uso

  • Sistema produtor: anexar arquivo à requisição.

Frequência: 5 vezes ao dia.

Precondições

  • Arquivo(s) presente na solicitação.

Sequência Típica

  1. CentralUFG valida arquivos enviados pelo sistema produtor, conferido:
  1. Formato do arquivo é de imagem reconhecida (Ex. png, jpg, gif, etc).
  2. Tamanho do arquivo está dentro do limite estabelecido.
  1. Se arquivos foram validados CentralUFG muda estado da requisição para Aceita e envia referência do arquivo, caso contrário - Fluxo Alternativo 01.
  2. CentralUFG responde código [0].
  3. Caso de uso se encerra.

Fluxo Alternativo 01

  1. CentralUFG muda estado da requisição para Negada. 
  2. Registra descricaoEstado da solicitação com o motivo da negação.
  3. CentralUFG responde código [7]
  4. Caso de uso se encerra.

Mur-Uso-ExecDivul – Executar divulgação

Descrição

A CentralUFG realiza ações apropriadas sobre as divulgações. A CentralUFG divulga as informações agendadas para a data corrente e encerra as divulgações ao fim do período que consta na solicitação.

Fontes de requisito

  • ReqSis14
  • ReqSis15
  • ReqSis16
  • ReqSis17
  • ReqSis18
  • ReqSis19
  • ReqSis20
  • ReqSis21

Atores, interesses e frequência de uso

  • Dispositivo android: Ser notificado quando houver alguma informação dirigida ao usuário vinculado ao dispositivo. Outro interesse é a disponibilidade da informação para consulta. Frequência: 1 vez ao dia.

Sequência típica

  1. CentralUFG busca todas as divulgações agendadas.
  2. CentralUFG executa divulgaPublicacao.
  3. CentralUFG executa divulgaNotificacao.
  4. CentralUFG busca todas as Publicações vigentes e data corrente após o período de vigência e as suspende.
  5. CentralUFG busca todas as Publicações agendadas e data corrente após o período de vigência e as registra como falha.
  6. CentralUFG busca todas as Notificações agendadas e data corrente após o período de vigência e as registra como falha.
  7. caso de uso se encerra.

divulgaPublicacao

  1. CentralUFG filtra as publicações.
  2. CentralUFG verifica se a data corrente esta no período de vigência da publicação e as coloca em vigência.
  3. retorna ao passo 4 da sequência típica.

divulgaNotificacao

  1. CentralUFG filtra as notificações.
  2. CentralUFG busca para cada notificação os dispositivos associados ao destinatário.
  3. CentralUFG solicita junto ao Serviço de mensagens nas nuvens, o envio das notificações para dispositivo android do usuário destinatário e conclui a notificação.
  4. retorna ao passo 4 da sequência típica

Mur-Uso-ComporHierPapeis - Compor hierarquia de GrupoDestinatario e  atribuição de papéis

Descrição

O sistema administrador informa à CentralUFG a estrutura de GrupoDestinatarios e os papéis a ser atribuído para determinados usuários. A Central UFG recebe os dados da hierarquia de papéis e aplica como uma nova estrutura de permissões.

Requisitos de qualidade

  • Funcionalidade: capaz de armazenar informações de forma segura (somente pessoas com autorização tenham acesso),  e precisa (completa e sem inconsistências).

Fonte do requisito

  • Termo de Abertura – Projeto Mural UFG
  • Requisitos de sistema: reqSis22, reqSis24

Atores, interesses e frequência de uso

  • Sistema administrador: Informar hierarquia de GrupoDestinatários e papéis que devem ser atribuídos aos usuários.
  • Frequência: 1 vez no semestre.

Precondições

  • Grupos de destinatários disponíveis.
  • Usuários com respectivos papéis disponíveis.

Sequência Tipica

  1. CentralUFG valida credenciais de usuário administrador.
  2. Central UFG verifica se as seguintes informações constam na solicitação:
  1. Dados de unidade.
  2. Dados dos cursos associados a uma unidade.
  3. Dados das turmas associados a um curso.
  4. Dados de destinatários associados a um usuário e a uma turma.
  5. Dados de docentes associados a um usuário e a uma turma.
  6. Dados de coordenadores associados a um usuário e a um curso.
  7. Dados de diretores associados a um usuário e a uma unidade.
  8. Dados de ouvidores associados a um usuário.
  9. Dados de reitores associados a um usuário.
  1. Central-UFG processa os dados da requisição, estruturando hierarquicamente GrupoDestinatario e os papéis ao usuário (ver Modelo de domínio) e persiste as informações.
  2. Cria um usuário para dispositivos sem vinculo com CAS. (veja Mur-Ent-Usu-Usuario no Modelo de domínio)
  3. Central responde código [20000]
  4. Caso de uso se encerra.

Mur-Uso-ValCredAdm - Validar credenciais de Administrador

Descrição

Um sistema adminsitrador gostaria de saber se a credencial de administrador é válida antes de solicitar qualquer solicitação adminstrativa. O sistema adminsitrador solicita a validação da credencial apresentada. A CentralUFG válida a credencial e responde a solicitação.

Fonte do requisito

  • ReqSis22
  • ReqSis23

Atores, interesses e frequência de uso

  • Sistema administrador: Um sistema adminsitrador gostaria de saber se a credencial de administrador é válida antes de solicitar qualquer solicitação adminstrativa.

Frequência: 2 vezes ao dia.

Sequência Tipica

  1. Sistema administrador solicita validação de credencial.
  2. CentralUFG válida credencial apresentada.
  3. CentralUFG responde código [20000]
  4. Caso de uso se encerra.
